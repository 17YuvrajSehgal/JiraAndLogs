## Part A — Project Briefing (presentation notes)

### A1. The 30-second pitch

The product is a **Jira-aware observability triage layer**. For every telemetry window (~60s slice of logs/metrics/traces/k8s state from one service), it decides:

1. **Is this ticket-worthy, borderline, or noise?** — triage classification, the primary task
2. **Which past Jira ticket is the closest historical match?** — retrieval, the product's main differentiator

The thesis: **historical Jira tickets are operational ground truth.** Mining them as a time-ordered "memory corpus" lets a model triage telemetry like a senior on-call engineer, with citations the engineer can act on.

The project is positioned **product first, research paper second** — both come out of the same evaluation harness (`src/comparison/`).

### A2. Why this matters commercially

| Current state in industry | Pain |
|---|---|
| Telemetry → static alert rules → on-call queue | Alert fatigue; missed incidents |
| Jira accumulates incident knowledge | Never fed back into alerting |
| Engineers manually grep past tickets during outages | Slow; depends on who was on-call last time |

No productized tool today does both (a) ticket-worthy vs noise decision AND (b) grounds it in past Jira evidence. That is the commercial whitespace.

### A3. Why a custom dataset (the hard part)

Public Jira datasets have tickets but no linked telemetry. Public observability datasets have telemetry but no linked Jira. To train a model that does both, the team has to *generate the paired data*. The repo's bulk is the data factory:

```
1. TELEMETRY LAYER (M0–M5)  → instrument 10 microservices across 5 langs
2. LAB LAYER                → kind/GCP + 27 fault scenario families
3. DATASET LAYER            → raw → ML-ready JSONL + time-ordered Jira memory
4. PRODUCT LAYER            → 12+ ML pipelines benchmarked on the same splits
```

### A5. Headline results (v5-quick, 1,020 windows, 13 families)

From `docs/weekly-report.md`:

- **Triage F1 at FPR=5%**: Calibrated Random Forest **0.43, ROC-AUC 0.88**; HGB/XGBoost-GPU close behind

- **Orphan recall gap**: every pipeline in `signal_learning` bucket (<10pt) — classifiers do not depend on memory match
- **LLM humanizer**: 3-4× BM25 retrieval lift just from rewriting Jira tickets to read like real on-call narratives

### Q1. How do we generate load?

**Source:** `microservices-demo-google/src/loadgenerator/locustfile.py`, `deploy/research-lab/scenarios/traffic-profiles/baseline-checkout-mix.yaml`

Load is generated by a **Locust-based loadgenerator** deployment running continuously inside the cluster as a regular Online Boutique service.

**Locust configuration** (`locustfile.py`):
- Class: `WebsiteUser(FastHttpUser)` with `wait_time = between(1, 10)` (each user pauses 1–10 seconds between requests)
- Synthetic users powered by the `faker` library (`fake.email()`, `fake.street_address()`, `fake.credit_card_number(card_type="visa")`)
- Hard-coded list of 9 product IDs (`0PUK6V6EV0`, `1YMWWN1N4O`, …) — load picks one at random
- Tasks dictionary with weighted probabilities:

| Task | Weight | What it does |
|---|---:|---|
| `index` | 1 | GET / |
| `setCurrency` | 2 | POST /setCurrency (random from EUR/USD/JPY/CAD/GBP/TRY) |
| `browseProduct` | **10** | GET /product/{id} (dominant) |
| `viewCart` | 3 | GET /cart |
| `addToCart` | 2 | GET /product/{id} + POST /cart |
| `checkout` | 1 | addToCart + POST /cart/checkout (full purchase with fake CC) |

**Traffic profile** (`baseline-checkout-mix.yaml`):
- `users: 25` concurrent virtual users
- `spawn_rate: 3` users/sec ramp-up
- `wait_time_seconds: { min: 1, max: 10 }`
- Expected primary paths: `/`, `/product/{id}`, `/cart`, `/cart/checkout`, `/setCurrency`
- Expected normal error rate: `very_low`; latency: `steady`

The same loadgenerator runs throughout the entire collection — baseline windows and fault windows. Faults are injected *on top of* this continuous traffic, not by changing the traffic pattern.

### Q2. How often do we inject issues?

**Source:** `deploy/research-lab/run-plans/dataset-v5-new-families-a.json` (representative plan), `scripts/research-lab/collect-dataset-plan.ps1`, `scripts/research-lab/run-scenario.ps1`

Faults are injected on a **deterministic schedule defined per run plan**, not at random intervals. The cadence:

**Within one run** (e.g. `dataset-v5-new-families-a.json`):
- Run plan is an ordered JSON list of scenario entries
- Each entry references a scenario YAML (baseline or fault) + duration_seconds + post_window_seconds
- Plans alternate **baseline → fault → baseline → fault → …**, e.g.:
  ```
  baseline-normal-traffic (90s)
  deploy-rolling-cart-graceful (60s)       ← fault
  redis-blip-30s-recovery (30s)            ← fault
  deploy-rolling-frontend-graceful (60s)   ← fault
  paymentservice-flake-recovers (45s)      ← fault
  frontend-1-of-3-restart (45s)            ← fault
  currency-api-blip-major (60s)            ← fault
  currency-timeout-recovers (30s)          ← fault
  cartservice-1-of-3-restart (45s)         ← fault
  deploy-canary-rollback-quick (45s)       ← fault
  recommendation-model-blip-minor (30s)    ← fault
  baseline-normal-traffic (90s)
  ```
- Between each scenario there's an exporter pass that pulls telemetry for the scenario's windows.
- `default_post_window_seconds: 75` defines the recovery observation period after each fault.

**Across runs in a corpus** (`deploy/research-lab/corpora/dataset-v5-large.json`):
- Corpus is a list of `{plan, repeat}` entries — e.g. `compact-a` repeats 14×, `new-families-a` repeats 11×, `control` repeats 8×.
- Total: 100 runs × ~10 scenarios/run = ~920 episodes, ~7,400 telemetry windows.

**Frequency cadence summary:**
- One fault per scenario, one scenario at a time, sequential.
- Faults are typically 30–600 seconds long (most are 30–240s; `cartservice-memory-leak-ticket-worthy` is 600s).
- Each fault is separated from the next by an export-telemetry-window pass + a post-window observation period (60–180s).
- A typical run completes ~10 scenarios in ~30–90 minutes.

### Q3. When and how are Jira issues collected?

**Source:** `scripts/research-lab/generate-shadow-jira-issues.ps1`, `src/jira_humanizer/llm_rewrite.py`, `docs/jira-shadow-issue-contract.md`, `schemas/jira_shadow_issue.schema.json`

**Step 1 — Synthetic shadow Jira generation (post-scenario):**

After each scenario finishes and telemetry has been exported, `generate-shadow-jira-issues.ps1` runs against the dataset run. Logic:

1. Reads `data/runs/<RUN>/episodes.jsonl` (one row per scenario invocation).
2. For each episode where `jira_candidate=true` AND `produces_jira_ticket != false` (D12 orphan scenarios skip this step), it constructs a Jira-shaped JSON record.
3. Pulls `telemetry_window_ids`, `trace_ids`, `alert_fingerprints` from the episode + associated windows/alerts.
4. Builds the issue with: `summary` (templated from severity + affected_services + incident_type + scenario_id), `priority` (from severity: critical→Critical, major→Major, minor→Minor), `components` (= affected_services), `labels` (production-safe: `incident`, `production`, `p0/p1/p3`, `customer-impact`).
5. Synthesizes a 3-event history (`Needs Triage → In Progress → Resolved`) with timestamps relative to `episode.start_time` + activity events (issue_created, priority change, comment, status change).
6. Crucially, the `telemetry_links` is a **sibling field** to user-visible fields (description, labels) so lab-only IDs never leak into the text.
7. Appends to `data/runs/<RUN>/jira_shadow_issues.jsonl`.

**Step 2 — LLM humanization** (`src/jira_humanizer/llm_rewrite.py`):

After all runs in a prefix complete, the humanizer rewrites every shadow ticket to read like a real on-call narrative (see Q17 for full details).

**Step 3 — Build time-ordered Jira memory corpus** (`scripts/research-lab/build_jira_memory_corpus.py`):

- Aggregates shadow issues across ALL runs matching a prefix.
- Computes `memory_text` = `"Summary: … | Components: … | Labels: … | Description: … | Comment 1: … | Comment 2: … | Comment 3: …"` (each comment capped 300 chars).
- Adds `available_as_memory_from` = issue's `created_at` (used for time-ordering at retrieval time).
- Sorts by `(available_as_memory_from, jira_shadow_issue_id)`.
- Writes to `data/derived/global/<GLOBAL_DATASET_ID>/jira-memory-corpus.jsonl`.

### Q4. How do we know we are creating the right Jira issue for the right fault injection?

**Source:** `deploy/research-lab/scenarios/faults/cart-redis-degradation-critical.yaml`, `scripts/research-lab/run-scenario.ps1`, `scripts/research-lab/generate-shadow-jira-issues.ps1`

Multiple layers ensure issue↔fault correctness:

**(1) Scenario YAML is the contract.** Each fault YAML carries an `expected_jira` block declaring exactly what the issue should look like — `issue_type`, `priority`, `components`, `labels`, `summary_template`, `description_template`. Example from `cart-redis-degradation-critical.yaml`:

```yaml
expected_jira:
  issue_type: Incident
  priority: Critical
  components: [cartservice, redis-cart, checkoutservice]
  labels: [synthetic-incident, dependency-failure, cartservice, redis, telemetry-linked]
  summary_template: "Cart dependency degradation caused checkout failures"
```

**(2) Episode carries provenance** (`run-scenario.ps1` line ~533–547):
- `incident_episode_id` = `<dataset_run_id>-<scenario_id>-<timestamp>` (unique, links episode to scenario unambiguously)
- `scenario_id`, `fault_id`, `traffic_profile_id`, `affected_services`, `severity`, `incident_type`, `root_cause_category` — all copied from scenario YAML
- `ground_truth.fault_type` from scenario YAML, used by retrieval ground truth
- `provenance.scenario_yaml_sha256` — the SHA256 of the scenario YAML file at episode-record time. A reviewer can detect any mid-collection YAML edit.
- `provenance.scenario_triage_block_present` — flag for whether a triage block exists in the YAML at that moment.

**(3) Jira generation is keyed by episode** (`generate-shadow-jira-issues.ps1` line 189–202):
- Iterates over episodes, filters by `jira_candidate=true`, looks up `episodeWindows` and `episodeAlerts` by `incident_episode_id`.
- Builds the issue with components = `episode.affected_services` and priority derived from `episode.severity`.
- Writes `incident_episode_id` into the Jira record itself, so each Jira row is permanently tied to one episode.

**(4) Update-back to episode** (line ~400–405): after the issue is created, `episode.jira_shadow_issue_id`, `episode.jira_issue_key`, `episode.telemetry_window_ids`, `episode.trace_ids`, `episode.alert_fingerprints` are written back into `episodes.jsonl`. So the linkage is bidirectional.

**(5) Validators** (`scripts/research-lab/validate-dataset-run.ps1`, `validate_l1_l2_telemetry.py`) enforce per-run completeness + per-feature distribution sanity.

### Q5. How do we know which telemetry/log window belongs to a given Jira issue and which window does not?

**Source:** `scripts/research-lab/build_window_memory_matchings.py`, `scripts/research-lab/triage_labels.py` (SCENARIO_FAMILIES, FAULT_TYPE_COMPATIBILITY)

**Direct linkage (within a run):**
- Each `telemetry_window` record carries `incident_episode_id` (set at scenario execution time, `run-scenario.ps1` line ~310–360).
- Each Jira shadow issue carries `incident_episode_id` (set at generation time).
- → matching windows = `telemetry_windows.jsonl WHERE incident_episode_id = <Jira>.incident_episode_id`. This is exact, not heuristic.

**Cross-run ground truth (the "is this window's matching past Jira X?" question)** is computed by `build_window_memory_matchings.py`. The matching logic (line 41–69):

```python
def compute_matches(window_start_time, window_run_id, window_family,
                    window_affected_service, window_fault_class, corpus):
    matched = []
    for entry in corpus:
        # OWN-RUN EXCLUSION: a window cannot retrieve Jira from its own run
        if entry.dataset_run_id == window_run_id: continue
        # TIME-ORDERING: Jira must precede the window
        if entry.available_as_memory_from >= window_start_time: continue
        # SCENARIO FAMILY must match
        if entry.scenario_family != window_family: continue
        # AFFECTED SERVICE must match
        if window_affected_service and entry.affected_service != window_affected_service: continue
        # FAULT COMPATIBILITY CLASS must match
        if entry.fault_compatibility_class != window_fault_class: continue
        matched.append(entry.jira_shadow_issue_id)
    return matched
```

A window "belongs to" a Jira issue iff it shares all four of: scenario family, affected service, fault-compatibility class, and the Jira pre-dates the window (and is from a different run). Fault-compatibility classes are defined in `triage_labels.py` `FAULT_TYPE_COMPATIBILITY` (collapses `pod_restart`, `pod_restart_nearmiss`, `service_restart`, `dependency_restart` etc. into a single `restart` class, etc.).

**Novelty + orphan logic** (line 142–185):
- `expected_in_memory` is True for `ticket_worthy` windows where `episode.produces_jira_ticket = True`.
- `expected_in_memory` is False for orphan scenarios (`produces_jira_ticket: false`) — ground truth forces `matches = []` and `is_novel = True` even if the family matches an existing Jira.
- `is_novel = True` for `ticket_worthy` windows where `compute_matches` returns nothing.
- Output: `data/derived/<RUN>/window_memory_matchings.jsonl`. Fields per row: `window_id, dataset_run_id, triage_label, scenario_family, affected_service, fault_compatibility_class, matched_memory_issue_ids, is_novel, expected_in_memory`.

This is the ground truth used by retrieval evaluation (Recall@k, MRR).

### Q6. When we train, what is the window size — how many lines and features of data do we feed?

**Source:** `scripts/research-lab/run-scenario.ps1` (lines 11–14 for window timings), `scripts/research-lab/triage_labels.py` (lines 455–534 for feature columns, lines 1010–1151 for evidence_text), `scripts/research-lab/build_triage_dataset.py`

**Time-window sizing** (from `run-scenario.ps1` defaults + run plan overrides):
- `PreWindowSeconds = 300` (5 min pre-fault baseline)
- `PostWindowSeconds = 180` (3 min recovery, can be overridden to 30/60/75 per plan)
- Active fault: defined by scenario `expected_duration_seconds` or plan's `duration_seconds` (range: 30–600s; long-running slow-leak uses 600s)

Each fault scenario produces **3 windows per affected service**: `pre_fault_baseline`, `active_fault`, `recovery_window`. Each baseline scenario produces **1 window per service**: `observation_window`.

### Q7. How do we inject faults — is it random?

**Source:** `deploy/research-lab/run-plans/*.json`, `scripts/research-lab/run-scenario.ps1` (lines 91–308 for action handlers), `deploy/research-lab/scenarios/faults/*.yaml`

**Fault injection is NOT random.** It is fully deterministic and reproducible:

- **Order of scenarios within a run** is fixed by the run plan JSON (`deploy/research-lab/run-plans/*.json`).
- **Repeat count per plan** is fixed by the corpus manifest (`deploy/research-lab/corpora/dataset-v5-large.json` `plans[].repeat`).
- **Injection mechanism** is one of 5 actions, defined per-scenario in `execution.action`:

| Action | Mechanism | Example scenario |
|---|---|---|
| `RecordOnly` | Sleep for duration; no actual change | `baseline-normal-traffic` |
| `SetEnv` | `kubectl set env deployment/<x> KEY=VALUE`, wait, restore | `productcatalog-bad-config-critical` |
| `RestartPods` | `kubectl delete pod -l <selector>`, wait for ready, sleep | `cartservice-1-of-3-restart`, restart scenarios |
| `ScaleDeployment` | `kubectl scale deployment/<x> --replicas=0`, sleep, scale back to `restore_replicas` | `cart-redis-degradation-critical`, outage scenarios |
| `ChaosMeshChaos` | `kubectl apply -f <chaos-mesh-manifest>` (NetworkChaos/StressChaos/DNSChaos), sleep, bounded delete with finalizer-patch fallback | D11 system-fault scenarios |

- Each scenario YAML carries its own action, target, parameters (e.g. `replicas: 0`, `restore_replicas: 1`, or `chaos_manifest: <path>`).
- The harness records `ground_truth.action_metadata` per episode capturing original replica count, original env values, chaos manifest path, etc. — for full reproducibility.

**Why deterministic?** Because the dataset is meant to be reproducible across collection runs, and because randomization would make per-family ground truth harder to construct.

What does vary across runs of the same plan:
- Exact timestamps (load is continuous, fault hits land at different wall-clock moments)
- Trace ID hashes (different request sequences)
- Some pod-level variation (which specific pod gets deleted)

### Q8. What is the ratio of normal traffic to fault injection?

**Source:** Each run plan JSON, weekly-report.md §4

Two ways to measure this:

**(a) Scenario count ratio** (per run plan):

| Plan                           | Baseline scenarios | Fault scenarios | Baseline % |
|--------------------------------|---:|---:|---:|
| `control-baseline-only`        | 6 | 0 | 100% |
| `dataset-v5-diverse-compact-a` | 2 | 8 | 20% |
| `dataset-v5-diverse-compact-b` | 2 | 11 | ~15% |
| `dataset-v5-new-families-a`    | 2 | 10 | ~17% |
| `dataset-v5-new-families-b`    | 2 | 6 | 25% |
| `dataset-v5-long-running`      | 2 | 2 | 50% |
| `dataset-v5-orphans`           | 2 | 8 | 20% |

The continuous load (Locust users) runs throughout — every scenario, baseline or fault, has the same 25-user traffic. The "baseline" scenarios are time slices during which **no fault is injected**.

**(b) Window-level label ratio** (the ratio that matters for ML):

From `docs/weekly-report.md` §3.1 (v5-quick test split): 450 windows, 96 ticket-worthy → **22% positive rate** for `ticket_worthy`. The rest is `borderline` + `noise`.

From `docs/dataset-v4-plan.md` `dataset-v4-large`: 22% borderline target, ≥15% hard-case target, rest noise. Triage contract acceptance gate requires "at least 20% of windows are noise and at least 5% are borderline".

**(c) Pre-fault baseline as built-in negative anchor:** every fault scenario emits a `pre_fault_baseline` window (5 min) before the active fault. That window is always labelled `noise`. So even fault scenarios contribute ~33% noise windows (1 baseline + 1 active + 1 recovery).

**(d) Control plan as false-positive anchor:** v5-large explicitly includes 8 pure-baseline runs (`control-baseline-only.json` × 8 = 48 baseline episodes × 6 services × 1 observation_window = ~288 pure-noise windows) to anchor the false-positive distribution.

### Q11. What were the major changes in the existing telemetry and what effect did it have on our dataset and why did we do it?

### Q12. How do we keep track of which Jira issue gets which label?

**Source:** `scripts/research-lab/triage_labels.py` (lines 398–452 for label record build, lines 27–38 for valid label vocab), `schemas/triage_window_label.schema.json`, `deploy/research-lab/scenarios/faults/cart-redis-degradation-critical.yaml`

**Per-window labels** are tracked separately from Jira issues. The window-label record (built by `build_triage_label_record`):

```
{
  telemetry_window_id, dataset_run_id, incident_episode_id, scenario_id,
  scenario_family, window_type,
  triage_label: "ticket_worthy" | "borderline" | "noise",
  triage_severity: "minor" | "major" | "critical" | null,
  triage_components: [<components>] | null,
  triage_reason_class: "outage" | "latency_regression" | "restart_with_impact"
                       | "bad_config" | "capacity" | "dependency_failure"
                       | "data_consistency" | null,
  is_hard_case: bool,
  rationale: <text>,
  source: "scenario_authored" | "human_adjudicated" | "derived",
  adjudicator: <name> | null,
  adjudicated_at: <timestamp> | null,
  derived_rule_id: "derive-from-scenario-fields-v1" | null,
  labels: {}
}
```

**Label source provenance (`source` field):**
- `scenario_authored` — taken directly from the scenario YAML's `triage.per_window.<window_type>` block. **Preferred.**
- `human_adjudicated` — written by `src/adjudication/adjudicate.py` after a human reviewed evidence and committed a label.
- `derived` — generated by the deterministic rule in `triage_labels.derive_triage_label` (uses scenario `jira_candidate`, `severity`, `window_type` fields). Default fallback when no YAML triage block exists.

**Per-Jira-issue labels** (the issue's priority, components, summary):
- Derived from `episode.severity` and `episode.affected_services` by `generate-shadow-jira-issues.ps1`.
- Stored on the Jira issue's `metadata` block (`priority, components, labels, status, resolution, ...`).
- The issue does NOT carry the triage_label per se — the triage label is a per-window property. But the issue maps to its episode, and the episode maps to multiple windows each with their own triage_label.

**Storage paths:**
- `data/runs/<RUN>/triage_window_labels.jsonl` — per-window label records (incl. provenance).
- `data/runs/<RUN>/jira_shadow_issues.jsonl` — per-issue records.
- `data/runs/<RUN>/episodes.jsonl` — the join key (`incident_episode_id`).

**Provenance integrity:** the episode record carries `provenance.scenario_yaml_sha256` so a reviewer can detect mid-collection YAML edits that would invalidate the `scenario_authored` source.

### Q14. What is the meaning of "window" and "scenario families" — do we inject all scenarios of the same type at the same time or is it injected randomly? What does "1020 windows and 13 families" mean?

**Source:** `README.md` §"Important terms", `scripts/research-lab/triage_labels.py` SCENARIO_FAMILIES, `scripts/research-lab/run-scenario.ps1` (window creation lines 310–362, 457–504), `docs/weekly-report.md` §4.1

**Window** = a labelled time slice of telemetry around an episode, *per service*. Defined in `run-scenario.ps1`:

| Window type | When | Default duration |
|---|---|---|
| `pre_fault_baseline` | Before the fault is injected, runs Locust traffic only | 300s (`PreWindowSeconds`) |
| `active_fault` | During fault injection | scenario `duration_seconds` (30–600s) |
| `recovery_window` | After fault stops, restoration phase | 60–180s (`post_window_seconds`) |
| `observation_window` | Used for baseline scenarios with `action=RecordOnly` | full scenario duration |

If a scenario lists `affected_services: [cartservice, redis-cart, checkoutservice]`, three windows are produced **per window type**: e.g., 3 services × 3 window types = 9 windows for one fault episode.

**Scenario family** = a semantic grouping of scenarios that share fault mechanism + affected service. Defined in `triage_labels.py` `SCENARIO_FAMILIES` dict (line 43). Example: `cart-redis` family contains 4 scenarios (`cart-redis-degradation-critical`, `redis-cart-restart-major`, `redis-cart-restart-nearmiss`, `redis-cart-intermittent-failure-major`). Why grouped: a model that learns "cart-redis fault" generalizes across these 4 specific scenarios.

**Injection order — NOT random:**
- Each run plan defines an ordered sequence of scenarios (typically baseline → fault → fault → … → baseline).
- Each scenario in turn injects exactly one fault sequentially.
- Across a corpus, plans are repeated N times in a fixed order (e.g. v5-large runs `control` 8×, then `compact-a` 14×, then `compact-b` 14×, etc.).
- **Scenarios of the same family are NOT batched together** — a single run interleaves multiple families. This avoids order-dependent confounding.

**"1020 windows / 13 families" = the v5-quick collected dataset** (`docs/weekly-report.md` §4.1):
- 16 dataset runs (controlled collection)
- 13 scenario families actually present (some D1 families never landed in v5-quick because of skipped scenarios; the v5-quick corpus manifest expected 22 but the actual collection landed 13)
- 146 incident episodes
- **1,020 telemetry windows** = sum of per-service × per-window-type across all episodes
- 48 Jira shadow issues
- ~56 million raw log lines on disk

(For comparison: v5-large projects ~7,400 windows across 27 families across 100 runs — a 7× scale-up.)

```
Summary: <metadata.summary>
Components: <metadata.components joined by comma>
Labels: <metadata.labels joined by comma>
Description: <metadata.description>
Comment 1: <first comment, capped 300 chars>
Comment 2: <second comment, capped 300 chars>
Comment 3: <third comment, capped 300 chars>
```

So when a retriever scores a window↔Jira pair, it considers:
- Summary line ("Customer checkout path seeing elevated failures")
- Components list ("cartservice, redis-cart, checkoutservice")
- Labels ("incident, production, p0, customer-impact") — note: lab labels like `dataset-*` and `scenario-*` are stripped by the humanizer
- Description (LLM-humanized, 1–3 sentences, first-person)
- Up to 3 comments (each by a different fictional reviewer, conversational tone)

- `LOG-EVENTS` section: up to 20 error/warning/dep_error log lines with structured key=value extraction (dep, op, method, status_code, err_class, peer_service, retry_attempt, latency_ms, kind)
- `TRACES` summary: `total_spans, error_spans, p50_ms, p95_ms` + top 5 root span names
- `K8S-EVENTS` section: up to 8 warning events for this service
- ID fields stripped (trace_id, span_id, EventId, request_id, session_id, correlation_id)
- No WINDOW header (removed because it leaked window_id substring)

### Q16. What metrics are we collecting and why are we using them, and what does a higher/lower value mean?

**Source:** `src/comparison/stratified.py` `_triage_metrics()`, `src/comparison/significance.py`, `src/comparison/stratified.py` `OrphanRecallGap` + `_gap_verdict`, `docs/triage-task-contract.md` §Metrics, `docs/weekly-report.md` §7

**Triage classification metrics** (per pipeline, per stratum):

| Metric | Definition | Higher means | Why we use it |
|---|---|---|---|
| **F1 @ FPR=5%** | Harmonic mean of precision and recall at the 5% false-positive operating point | Better balanced quality at the deployed operating point | Default headline. Corporate stakeholders expect a single number. >0.4 = useful. |
| **PR-AUC** | Area under precision-recall curve; threshold-free | Better quality across all thresholds, robust to class imbalance | Best metric when classes are imbalanced (22% positive). Headline for ML-savvy audience. |
| **ROC-AUC** | Probability a positive is scored above a negative; threshold-free | Better ranking quality (0.5=random, 1.0=perfect) | Sanity check; less informative on imbalanced data than PR-AUC. |
| **Precision @ FPR=1% / 5%** | Of windows flagged at this FPR, fraction truly ticket-worthy | Fewer false pages | Models the "alert quality" the on-call experiences. |
| **Recall @ FPR=1% / 5%** | Of truly ticket-worthy windows, fraction caught at this FPR | Catches more incidents | Models "what fraction of real tickets we'd surface". |
| **ECE** (Expected Calibration Error) | Average gap between predicted probability and observed positive rate, binned | Lower (closer to 0) = better calibrated | Required because P@FPR=1% is meaningless if probabilities aren't calibrated. |
| **Cost-weighted F-beta (β=2)** | Like F1 but weights recall 2× more than precision | Better recall-leaning quality | Encodes the cost asymmetry: missing a real incident is worse than one false alarm. |
| **n_pos / n_neg** | Sample sizes by class per stratum | — | Sanity for stratified rows with tiny n. |

**Retrieval metrics** (per pipeline, on `ticket_worthy` windows with a non-empty `matched_memory_issue_ids` ground truth):

| Metric | Definition | Higher means | Why we use it |
|---|---|---|---|
| **Recall@1** | Of windows with a matching past Jira, fraction where the right one is #1 | Better top result quality | "If the on-call only reads the first suggestion, do we get it right?" |
| **Recall@3** / **@5** | Same, in top-3 / top-5 | Better top-k coverage | Default product number — the UI shows 5 suggestions. |
| **MRR** (Mean Reciprocal Rank) | Average of 1/rank of the correct issue | Higher (max 1.0) = answers near rank 1 | Single-number summary of ranking quality. 1.0 = always rank 1; 0.5 = always rank 2; 0.33 = always rank 3. |
| **Novelty F1** | Quality of detecting truly novel windows (no matching past ticket) | Better novel-incident detection | Critical for "this is a new kind of incident" alerts. |
| **n_retrievable** | Sample size of retrievable windows | — | Sanity for retrieval rows. |

**Generalization metrics:**

| Metric | Definition | Higher means | Why |
|---|---|---|---|
| **LOFO macro PR-AUC** | Train on N-1 families, score on held-out family, average across folds | Better generalization to unseen scenario shapes | Strongest single test that the model isn't overfit to the test families. |
| **LOFO macro ROC-AUC** | Same for ROC-AUC | Same | Compare against PR-AUC on the same fold. |
| **Folds** | Number of LOFO folds | — | Sanity — single-class folds skipped. |

**Orphan recall gap metric** (`src/comparison/stratified.py` `compute_orphan_recall_gap`, `_gap_verdict`):

| Quantity | Definition |
|---|---|
| `n_reported` | Count of ticket_worthy windows with `expected_in_memory=True` |
| `recall_reported` | Of those, fraction the pipeline flagged as `ticket_worthy` |
| `n_orphan` | Count of ticket_worthy windows with `expected_in_memory=False` |
| `recall_orphan` | Of those, fraction the pipeline flagged as `ticket_worthy` |
| `gap_pts` | `100 × (recall_reported − recall_orphan)` |
| `verdict` | `"signal_learning"` if gap<10 (good — model uses telemetry), `"borderline"` if 10–20, `"pattern_matching"` if >20 (bad — model just memorizes Jira) |

Lower gap = the model generalizes from telemetry signal rather than depending on memory match.

**Statistical reliability metrics** (`src/comparison/significance.py`):

| Metric | Definition | Use |
|---|---|---|
| **Paired bootstrap 95% CI** | Resample test windows 1,000–10,000 times, recompute both pipelines' metrics on the resample, report `[lo, hi]` of the delta | Whenever you claim "X beats Y" — if CI of (X−Y) crosses zero, the difference is not statistically significant |
| **Pairwise significance (p-value)** | Two-sided p-value of "delta != 0" from the bootstrap distribution | The "Does Jira help log analysis?" headline is built from these |

### Q17. How did we use LLMs to humanize the JIRA tickets?

**Source:** `src/jira_humanizer/llm_rewrite.py`, `src/jira_humanizer/rewrite.py`, `docs/weekly-report.md` §2

**Why:** the synthetic shadow Jira issues, even after the template-only humanizer, read like obviously synthetic data. A stakeholder opening any sample ticket would see scenario IDs, trace IDs, and lab metadata — instant credibility problem. The LLM humanizer rewrites each ticket to read like an actual on-call engineer wrote it.

**Backend:** Qwen 2.5 Coder 14B served by **LM Studio** (OpenAI-compatible API on `http://localhost:1234`). Cost: ~5s per issue. v5-quick (48 issues): ~4 min. v5-large (~430 issues): ~50 min.

**Pipeline** (`llm_rewrite.py` `_humanize_issue_with_llm`):

1. For each shadow Jira issue, extract from `metadata`: `components`, `priority` (→ severity), `summary` (as hint), primary service (`components[0]`).
2. Determine `scenario_family` from a `scenario-<id>` label or default to `recovered-in-window`.
3. Look up `_FAMILY_BRIEFS[family]` (per-family English brief, ~22 families covered) — e.g., `cart-redis: "the cart store backend (Redis) is unhealthy — connection timeouts or intermittent failures; customers see cart-add errors and lost carts"`.
4. Build the chat prompt:
   - **System**: *"You are an on-call site reliability engineer writing a real Jira ticket during an incident. Respond with JSON only — no prose, no markdown fences. NEVER mention 'synthetic', 'lab', 'research', 'dataset', 'scenario', or any identifier that looks like a UUID, a timestamp, or a numeric run id. Write the way an engineer would actually type into Jira at 2am: short, direct, first-person where it fits."*
   - **User**: incident context (severity, services, primary service, family brief, summary hint) + instructions for JSON output with `description` (1-3 sentences first-person) and `comments` (exactly 3 strings, simulating 3 different team members across the lifecycle).
5. Parse the JSON response (with `__MD_FENCE_RE` to strip ```json fences if Qwen adds them).
6. If LLM fails or returns malformed JSON, fall back to the deterministic **template humanizer** in `rewrite.py`.
7. Apply the template humanizer's label cleanup + name pool, then **swap in LLM-generated text**:
   - Replace `description` field
   - Assign 3 fictional names from `_NAMES` pool (Sarah Chen, Marcus Williams, Priya Patel, Diego Hernandez, …, 26 names total) deterministically by SHA-256 of `episode_id + "llm-comment-authors"`
   - Build conversational `comments_body` with names + timestamps spaced 18 min apart starting 6 min after `metadata.created_at`
   - Rewrite the activity events of type `"comment"` with the new author/body/timestamp

**Example sample output** (from `docs/weekly-report.md` §2):
> *"Checkout service pods are crashing repeatedly, causing high latency and failed transactions on the frontend. Panic was related to database connection timeouts after the load balancer changes; we reverted the driver and traffic stabilized."*

**Backup:** original `jira_shadow_issues.jsonl` is copied to `jira_shadow_issues.jsonl.pre-humanize.bak` before in-place rewrite.

**Measured effect** (`docs/weekly-report.md` §1, §2):
- BM25 retrieval R@5: **0.090 → 0.269** (3× lift)
- BM25 MRR: **0.056 → 0.181** (3× lift)
- Ensemble F1: +4pt; ROC-AUC: +3pt

The 3× retrieval lift confirms the humanizer is not just a credibility fix — semantically richer ticket prose actually carries more retrieval signal.

### Q18. How will human adjudication help with this project when in production?

1. Reviewer picks a borderline or hard-case window via `--next` or `--window <id>`.
2. The tool **blinds** `scenario_id`, `scenario_family`, and all `triage_*` fields from the reviewer's view — they only see telemetry, not the lab's authored label.
3. Renders 4 evidence streams to the reviewer:
   - Triage features incl. delta-from-baseline
   - Loki log summary with sample error lines
   - Tempo span summary with top error span names
   - Jira-memory retrieval hits (summary + memory_text slice)
4. Prompts the reviewer for `{label, severity, components, reason_class, rationale}`.
5. Writes a schema-conformant row to `data/derived/<run-id>/triage_window_labels.jsonl` with `source: human_adjudicated`, `adjudicator: <name>`, `adjudicated_at: <utc-iso>`.
6. After commit, reveals the scenario's authored truth label for calibration spot-check.

**Pending workload:** 720 borderline + 1,312 hard-case windows queued for review (per `dataset-todo.md` "Pending — Needs external resources").

**Planned D0.3 — LLM-as-first-reviewer:** Pipe the rendered evidence to Claude, capture machine-suggested label as a separate provenance source (`source: llm_suggested`). Makes 100% adjudication feasible at scale.

**Why human adjudication matters in production:**

| Production role | Benefit |
|---|---|
| **Borderline calibration** | The product reports a 3-class decision (ticket_worthy/borderline/noise). Borderline is by definition where reasonable engineers disagree. Inter-rater reliability from adjudication gives a calibration ceiling we can't exceed. |
| **Ground truth refinement** | The first model is trained on scenario_authored labels (the scenario author's intent). Engineers reviewing live data may disagree with that intent — and they would be right, because production engineers know the org's tolerance. |
| **Feedback loop** | Phase 2 product surface: engineers click "agree / disagree / wrong service / wrong severity" on the model's verdict. Each click becomes a new human_adjudicated label that retrains the next model version. |
| **Audit trail** | For compliance (SOX, SOC2), every label with a human reviewer + timestamp is recorded. Required when the product is in the on-call loop. |
| **Drift detection** | If the rate of human disagreement with the model rises over time, the model is drifting — trigger retraining. |
| **Hard-case curation** | The 1,312 hard cases are the model's weakest points. Adjudicated truth here yields the biggest per-row signal lift. |
| **D0.3 LLM bootstrap** | LLM-suggested labels (cheap) become first-pass; human reviewers focus only on LLM-vs-scenario disagreements. Saves >80% of human time while keeping a human in the loop. |

The triage contract demands `human_adjudicated` for every borderline + hard-case window before publishing as v5 (`docs/triage-task-contract.md` §"Label Sources" — "For v4, every borderline and is_hard_case window must be human_adjudicated").

### Q19. How does the system make sure it is replicating production-like scenarios and not making the product look good?

**Source:** `docs/telemetry-implementation-decisions.md` §M0.6, `README.md` §"Production-realism discipline (bias-avoidance)", `microservice-changes.md`, `scripts/research-lab/validate-run-feature-distribution.ps1`

The team explicitly committed to **9 documented divergences** from production (lifted verbatim into the v5 dataset README at publish time). The intent is: tell the user what is unrealistic so the model's lift can be honestly attributed.

**9 known divergences:**

1. **Trace sampling at 100%** (`AlwaysSample`). Production typically samples 1–10% head-sampled or tail-sampled on errors/latency. Dataset density traded against sampling realism; models trained here may over-rely on span fan-out features.
2. **Single-cluster, single-region.** All services run in one kind cluster (dev) or one GKE cluster in one region (cloud). No cross-region failures, no inter-cluster partitions, no multi-AZ failover.
3. **No real customer PII.** loadgenerator emits synthetic users; paymentservice's `simple-card-validator` matches `visa`/`mastercard` patterns only.
4. **Synthetic traffic patterns.** No diurnal cycle, no weekly cycle, no business-hour effect (Phase D8 will address in v5.1 or v6).
5. **Single fault per run.** Real incidents often have multiple correlated faults. Cascade scenarios (Phase D5) partly address this; full multi-fault correlation is out of scope for v5.
6. **Shadow Jira issues, not engineer-written.** All Jira rows are generated from scenario YAML templates by `generate-shadow-jira-issues.ps1`. Real tickets carry inconsistencies, typos, out-of-band signals (Slack, postmortems, manager pressure) that synthetic issues don't. (Mitigated by LLM humanizer, but still synthetic.)
7. **No alert-fatigue baseline.** Real on-call queues have a background level of false positives. v5's `post-deploy-churn` family partly addresses this.
8. **No deploy event correlation in v4/v5.** Phase D8 will add synthetic deploy events with correlated logs/metrics in v5.1 or v6.
9. **No GPU-served services.** Recommendation/ad/fraud often use GPU-backed inference in real e-commerce; Online Boutique's `recommendationservice` uses a Python sort-by-popularity proxy. Affects which failure modes are realistic.

**Active anti-bias enforcement** is in the M0–M5 telemetry rules (`microservice-changes.md`, `README.md` §"Production-realism discipline"):

- **Don't** add log fields, span attributes, or metric labels that name our scenarios (`scenario.id`, `fault.injected`, `expected_severity`).
- **Don't** label metrics by anything from `scripts/research-lab/triage_labels.py` (no `scenario_family`, `fault_type`).
- **Don't** propagate `dataset_run_id` through application baggage.
- **Don't** include any field listed as eval-only in `docs/triage-task-contract.md` "Field Policy".
- **Do** add only what a real SaaS company in 2026 would emit anyway.

**Automated enforcement** (`scripts/research-lab/validate-run-feature-distribution.ps1`, `validate_run_feature_distribution.py`):
- After every run, the **leakage canary** checks that no feature column is perfectly correlated with `scenario_id`, `scenario_family`, or `triage_label`. If any column is, the run fails validation.
- M5.2e leakage canary on r04-followup: PASS, 0 fails, 4 warns.

**Evidence-text scrubbing** (`triage_labels.py evidence_text_from_raw`):
- `WINDOW <id>` header line removed (line 1043 comment): "WINDOW header — that line embedded the lab-only window_id / scenario_id / window_type as substring tokens, which lexical and embedding pipelines were silently using as label proxies."
- Per-request IDs stripped from log bodies: `trace_id, span_id, EventId, parent_span_id, request_id, session_id, correlation_id`.
- Production-safe field allowlist: `dep, op, method, status_code, err_class, peer_service, retry_attempt, latency_ms, kind`.

### Q20. What production-fidelity disciplines have been used to increase credibility?

**Source:** Same as Q19 + `docs/triage-task-contract.md` §"Field Policy", `scripts/research-lab/triage_labels.py` `_summarize_log_body`, `src/jira_humanizer/rewrite.py` header

**Eval-only fields (the field-policy contract):** No production-facing model is allowed to read:
- `scenario_id`, `incident_type`, `root_cause_category`
- `severity` (lab-authored), `jira_candidate` (lab-authored)
- `triage_label`, `triage_severity`, `triage_components`, `triage_reason_class` (these are the targets)
- `is_hard_case`
- Anything under `ground_truth`

Allowed inputs: logs, metrics, traces, Kubernetes events, deployment state, alert state, deterministic features derived from them.

**Disciplines layer by layer:**

| Layer | Discipline |
|---|---|
| **Telemetry instrumentation** | No log/metric/span field named after our scenarios. Tested via leakage canary. |
| **Window features** | `numeric_features_from_raw` only reads `raw/loki`, `raw/prometheus`, `raw/tempo`, `raw/kubernetes` — never any label field. M0–M5 supplement (`m05_*` columns) is per-service RED metrics, not scenario tags. |
| **Window evidence text** | `evidence_text_from_raw` strips per-request IDs, removes the WINDOW header line, keeps only production-realistic structured fields. Capped at 4000 chars to mimic real log-budget. |
| **Jira generation** | `generate-shadow-jira-issues.ps1` (lines 256–276): labels strip `dataset-*`, `scenario-*`, `synthetic-incident`, `telemetry-linked`. Telemetry-link fields (window_ids, trace_ids, alert_fingerprints) live in the `telemetry_links` SIBLING field, NOT in description/comments/labels. That sibling field is the retrieval ground truth and is invisible in production Jira. |
| **Jira humanization** | LLM system prompt explicitly bans the words "synthetic", "lab", "research", "dataset", "scenario", UUIDs, timestamps, numeric run ids. Description/comments fully rewritten by Qwen 14B; fictional human names assigned. |
| **Splits** | `triage-split-manifest.json` splits by **scenario family**, not by run. Per-run holdout is too easy because fault signatures repeat across runs of the same family. |
| **LOFO** | Leave-one-family-out macro reported for every pipeline — held-out family is one the model has never seen. PR-AUC 0.82 on this is the headline generalization number. |
| **Time-ordering** | At retrieval time, `compute_matches` (window memory matchings) only allows Jira entries with `available_as_memory_from < window.start_time` — the model cannot retrieve from the future. |
| **Own-run exclusion** | `compute_matches` excludes any Jira from the same `dataset_run_id` as the window. Prevents intra-run memorization. |
| **Validators** | Per-run validator + leakage canary on every collected run. Failed runs halt the corpus. v5-large promotion gate: `validate-dataset-run.ps1` (0 errors) + `validate-run-feature-distribution.py` (0 anomalies). |
| **Provenance** | Each episode carries `provenance.scenario_yaml_sha256` so reviewers can detect mid-collection YAML edits. |
| **9-divergence disclosure** | Published verbatim with the dataset. Reviewers can see what the dataset doesn't model. |

### Q22. How do we make sure it is not memorizing incidents?

**Source:** `docs/triage-task-contract.md` §"Split Rules", `src/comparison/pipelines.py` `lofo_evaluate`, `scripts/research-lab/validate_run_feature_distribution.py`, `scripts/research-lab/build_window_memory_matchings.py`, `src/comparison/stratified.py` `OrphanRecallGap`

**Six independent safeguards** against memorization:

**(1) Splits by scenario family, not by run** (`docs/triage-task-contract.md` §"Split Rules"):
> Per-run holdout (the ranking benchmark's split rule) is too easy for triage because a model can memorize fault signatures across runs of the same family.

`triage-split-manifest.json` assigns whole families to train/val/test. If `cart-redis` is in test, then *every* `cart-redis-*` scenario (4 of them) is in test — none leak into train.

**(2) Leave-one-family-out (LOFO) macro evaluation** (`src/comparison/pipelines.py` `_NumericClassifierPipeline.lofo_evaluate`):
- For each family in the corpus, train on the other 13–26 families, score on the held-out family.
- Report `macro PR-AUC = mean across folds`.
- v5-quick result: **RF 0.82 macro PR-AUC, HGB 0.82, logistic 0.62**. RF/HGB hold up to never-seen families; logistic collapses → the strong models learn telemetry SHAPES, not scenario IDs.

**(3) Leakage canary** (`scripts/research-lab/validate_run_feature_distribution.py`, run after every collection):
- Verifies no feature column is perfectly correlated with `scenario_id`, `scenario_family`, or `triage_label`.
- v5-large promotion gate requires 0 canary fails.
- M5.2e check on r04-followup: PASS (30 rows, 0 fails, 4 warns).

**(4) Production-safe field policy enforcement** (`docs/triage-task-contract.md` §"Field Policy"):
- Models are forbidden from reading `scenario_id`, `severity` (lab-authored), `jira_candidate`, the triage targets, `is_hard_case`, anything under `ground_truth`.
- Allowed inputs: logs, metrics, traces, k8s events, deployment/alert state, deterministic features.
- Evidence text scrubbing (Q15) strips per-request IDs and the WINDOW header so substring-leak via tokens is impossible.

**(5) Time-ordering + own-run exclusion at retrieval** (`build_window_memory_matchings.py compute_matches`):
- A window can ONLY retrieve Jira issues with `available_as_memory_from < window.start_time` (no peek into the future).
- A window CANNOT retrieve Jira from its own `dataset_run_id` (no intra-run memorization).

**(6) Orphan recall gap as adversarial test** (`stratified.py compute_orphan_recall_gap`):
- 192 orphan ticket_worthy windows in v5-large (32 in v5-quick) have no matching Jira at all.
- If a pipeline's `recall_orphan` collapses vs `recall_reported`, the pipeline depends on Jira lookup — it's pattern-matching, not signal-learning.
- v5-quick result (`docs/weekly-report.md` §3.4): every pipeline lands in the `signal_learning` bucket (gap < 10pt). None depend on memory match to identify ticket-worthiness.

**Auxiliary safeguard:** Jira humanization (Q17) explicitly forbids "synthetic", "lab", "research", "dataset", "scenario", UUIDs, timestamps in user-visible fields. Strips lab labels (`dataset-*`, `scenario-*`).

### Q23. What models and approaches are being used?

**Category 1 — Pure numeric classical ML** (treats triage as tabular classification on the 86 numeric features):

| Pipeline name | Underlying model | Notes |
|---|---|---|

| `calibrated_random_forest_numeric` | sklearn `RandomForestClassifier` + `CalibratedClassifierCV(method='isotonic', cv=3)` | `n_estimators=300, min_samples_leaf=5, class_weight=balanced`. Calibration matters for Precision@FPR. **Leader on v5-quick.** |

**Category 3 — Pure retrieval (rank past Jira given window evidence):**

| Pipeline name | Backend | Notes |
|---|---|---|
| `bm25_retrieval_only` | BM25 over tokenized memory_text | Cheap baseline. No GPU. v5-quick R@5=0.269. |
| `nomic_retrieval_only` | Dense embed via Nomic `text-embedding-nomic-embed-text-v1.5` (768-dim) over LM Studio + cosine similarity | **2× lift over BM25** (R@5=0.527). Fail-soft to BM25 if LM Studio unreachable. |
| `nomic_then_lm_rerank` | Nomic top-10 pool + Qwen 2.5 Coder 14B reranks via chat completion | v5-quick: **hurts** R@5 (0.527 → 0.417). Keep for v5-large where larger corpus may have more near-duplicates that reranker can disambiguate. |

**Category 4 — Neural fusion (text + numeric in one model):**

| Pipeline name | Architecture | Notes |
|---|---|---|
| `bi_encoder_hybrid` | `sentence-transformers/all-MiniLM-L6-v2` (384-dim) on `evidence_text` → concat with 28 numeric → `LogisticRegression` head | GPU-aware (RTX 5060 sm_120 detected via `util/device.py`). Mid-pack F1 on v5-quick — needs more data + fine-tuning. |

**Category 5 — Ensemble:**

| Pipeline name | Method | Notes |
|---|---|---|
| `ensemble_mean` | Mean of all pipelines' calibrated scores | **Best retriever overall on v5-quick** (R@5=0.571, MRR=0.392). |

**Category 6 — MemoryGraph (agentic cross-context retrieval, newest, commit `3a9e28f`):**

| Pipeline name | Architecture | Notes |
|---|---|---|
| `memorygraph` | Typed entity graph over window + Jira; rule-based or LLM-planned skill-chain agent | Adds structured reasoning + match explanation. |
| `memorygraph_hybrid` | `memorygraph` + `NumericBlendSkill` (HGB head on numeric features) | Blends per-window numeric prob into per-candidate graph score. |
| `memorygraph_full` | `memorygraph_hybrid` + Nomic dense similarity | Falls back to BM25 if LM Studio unreachable. |

**Backends used across pipelines:**
- Classical: sklearn, xgboost (CUDA)
- Embeddings: Nomic `text-embedding-nomic-embed-text-v1.5` via LM Studio
- Chat / rerank: Qwen 2.5 Coder 14B via LM Studio
- Neural encoder: sentence-transformers `all-MiniLM-L6-v2`
- Retrieval lex: BM25
- Log mining: Drain-lite template miner

**Train/eval contract:** every pipeline uses the same `target_fpr=0.05` for threshold tuning on validation, same train/val/test splits from `triage-split-manifest.json`, same paired-bootstrap CI for pairwise comparison.
