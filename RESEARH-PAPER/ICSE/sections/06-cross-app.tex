\section{Cross-App Generalization}
\label{sec:cross-app}

The cascade and its dataset are constructed on a single application. A natural question is whether the system transfers to a structurally different one without modification. This section reports the experimental setup we use to answer it, the two evaluation columns we report against (a strict zero-shot column and an L1-retrained column), the architectural axes along which the new application differs from the source, and an honest finding about the cascade's structural-versus-value feature compatibility.

\subsection{Experimental setup}
\label{sec:cross-app:setup}

The target application is the OpenTelemetry Demo (Astronomy Shop)~\cite{otel-demo}, a CNCF-hosted polyglot microservice benchmark that the OpenTelemetry community itself maintains as a reference implementation of native telemetry instrumentation. Relative to the source application from \S\ref{sec:approach:dataset}, the target differs along four orthogonal architectural axes. \emph{Service count and language diversity}: fifteen-plus services written in twelve-plus languages, against ten services in six languages on the source. \emph{Asynchronous messaging}: a Kafka broker carries order events from the checkout service to two downstream consumers (an accounting service and a fraud-detection service), introducing a non-RPC dependency boundary that the source application does not have. \emph{Trace depth}: typical request traces span roughly twice the depth of the source's, owing to the Kafka producer-consumer hand-off and a frontend-proxy layer in front of the user-facing service. \emph{Language-model-backed service}: a dedicated service performs an LLM inference inside the request path, introducing a failure mode (rate limiting, slow inference, malformed output) that has no source-application analogue.

The experimental protocol holds everything that defines the cascade fixed and varies only the upstream workload. The dataset construction harness, the telemetry-collection pipeline, the fault-injection orchestrator, the feature builder, the per-pipeline trainers, the cascade composition, and the evaluation script all run against the new application unchanged. We add two artifacts that the new application requires: a service-name catalog file that lists its canonical service short-names (used by the rule-based and language-model extractors so neither invents names the application does not have), and a scenario taxonomy that covers both the transferable failure modes (service outages, latency injections, pod restarts) and the new failure modes specific to the Kafka layer (broker outage, consumer lag, consumer crash, partition rebalance, dead-letter spike). The taxonomy adds five Kafka-specific families to the twenty-seven transferred from the source, for a total of thirty-two single-fault families.

The cascade is then reported against the new dataset under two evaluation columns. The \emph{zero-shot} column locks every component of the cascade---the Layer-1 triage stacker, the Layer-2 retrieval set and weights, the Layer-3 learned novelty classifier---to its source-application fit and applies the locked artifacts to the new application unchanged. This is the strictest test of external validity: nothing is retrained. The \emph{L1-retrained} column releases only the Layer-1 stacker, refitting it on the new application's training split via the same out-of-fold cross-validation used on the source; everything else stays locked. The motivation for the second column is operational: refitting a small linear stacker on a per-application training set takes seconds, while refitting the bi-encoder or the agent's verification stage is hours, so the L1-retrained column is the more representative number for a team deploying the cascade on its own workload.

A local-cluster pilot validated the harness end-to-end before the full collection: every fault-injection primitive emitted the expected telemetry, the global aggregation step produced a balanced label distribution across the pilot scenarios, and the feature builder ran to completion against the new application's telemetry. The headline cross-application numbers in \S\ref{sec:cross-app:headline} are computed on the full collection that the pilot unblocks.

\subsection{Graded difficulty: from single fault to compound anomaly}
\label{sec:cross-app:graded}

The target-application collection is, by design, strictly richer than the source's along one axis: every scenario family is exercised at four difficulty levels rather than one. Level~one is a single isolated fault, matching the source-application protocol and supplying the comparison column above. Level~two runs two independent faults concurrently in the same time window, so that more than one root cause is present at once and a system that returns a single top-one cannot fully resolve the page. Level~three runs two faults in a deliberate cascade in which the first fault induces the second after a brief emergence delay, so that the surface symptom (the downstream service breaking) is a derived effect of the upstream service's underlying failure. Level~four compounds an active fault with a slow saturation drift, so that the window's evidence is the superposition of two qualitatively different timescales. The levels are listed in increasing order of expected diagnostic difficulty.

\subsection{Kafka families: architectural-distance evidence}
\label{sec:cross-app:kafka}

The five Kafka-specific families exist in the target application because of its broker-mediated asynchronous communication, and have no source-application analogue. Their symptoms are qualitatively distinct from the RPC-based outages that dominate the source taxonomy: the failure surface for a Kafka broker outage manifests as producer-side write errors at the checkout service \emph{and} a near-silent consumer side that simply stops processing, rather than as the symmetric request-failure spike an RPC outage produces. Consumer-lag families exhibit no error-rate spike at all; their signature is an accumulating queue depth that becomes visible only at the next layer of the metric stack. \subsection{Structural compatibility and the feature-population gap}
\label{sec:cross-app:feature-gap}

The cascade's structural compatibility with the target application is complete: every artifact the source-application pipeline produces---scenario-labeled telemetry windows, the ninety-four-column numeric feature vector per window, the gold-linking structure between windows and Jira tickets, the per-pipeline prediction files that the cascade composes---is produced unchanged on the target application by the same code, and the feature-builder validation on a representative subset of target-application windows produces ninety-four out of ninety-four columns in the expected schema.

The cascade's \emph{value} compatibility is partial, and the gap is informative. Of the ninety-four feature columns, twelve populate identically from the OpenTelemetry SDK signals that both applications share (trace span counts, trace latency percentiles, container CPU and memory utilization, request-error counters derived from spans). Eighty-two of the ninety-four columns are application-specific in their value provenance: they depend on metric names that the source application's instrumentation emits (the project's own per-handler error counters, per-cart-operation success rates, per-dependency latency histograms) and on log-label conventions that the source application's logging pipeline produces (an \texttt{app=<service>} label that the target's collector does not emit by default). These eighty-two columns are populated as zero on target-application windows under the source-application configuration.

This is an honest cross-application generalization finding, not a defect of either application. The cascade's structural template is portable; its feature-value provenance is application-specific. In practice, an operator deploying the cascade on a new application has two productive choices. The cheap choice is to retrain the Layer-1 triage stacker on the new application's training split (the L1-retrained column above), which lets the stacker downweight the zero-valued columns automatically. The expensive choice is to extend the feature-builder's metric and label allowlists for the new application, recovering the full value provenance at the cost of a per-application instrumentation pass. Both options leave the cascade's retrieval and novelty layers, and the upstream pipelines that feed them, untouched. The structural-versus-value distinction is what makes the L1-retrained column useful: it isolates the architecture-portable parts of the cascade from the value-population concerns that an instrumentation difference exposes.
