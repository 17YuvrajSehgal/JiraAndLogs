\section{Threats to Validity}
\label{sec:threats}

\subsection{Internal validity}
\label{sec:threats:internal}

The primary internal-validity concern in a synthetic-but-labeled dataset is label leakage---the possibility that the telemetry surface inadvertently encodes the scenario taxonomy the model is supposed to learn from the joint distribution. We mitigate this with a discipline applied at dataset-construction time and enforced by code review: no telemetry field may name our scenario taxonomy, no log key may carry a fault-type string, no span attribute may identify the fault-injection run, and no alert name may reference a family the engineer would have to discover from symptoms. The discipline makes the data harder to debug at construction time but is what allows the evaluation to describe the production task rather than a fictional easier surrogate. A second internal-validity concern is that the Layer-1 stacker's coefficients reflect a model that has seen its own training data. We address this with five-fold stratified cross-validation: every training-fold window is scored by a stacker that did not observe it during fit, and the in-distribution headline numbers are computed against out-of-fold predictions. Language-model calls run at temperature zero with a fixed seed, and bootstrap confidence intervals use a fixed seed across pipelines so that paired differences are computed on identical resamples.

\subsection{External validity}
\label{sec:threats:external}

The lab differs from production along several axes that bound external generalization. We sample one hundred percent of traces, where a typical production cluster samples between one and ten percent; downsampling could in principle erode the trace-based signals the cascade depends on, although the dense retriever and the language-model agent operate primarily on log-derived text and would be less affected. We run a single cluster in a single region, so the cascade has never been exposed to multi-region failure modes (cross-region latency, partial-region outage, control-plane partition). The Jira corpus is synthetically generated by a humanizer rather than written by engineers, so it carries the humanizer's lexical fingerprint; we anchor the humanizer's length, code-block, and comment-density distributions to the real-Jira distribution from a public open-source corpus to limit the divergence. Each fault-injection run on the source application carries a single fault by construction; the target-application collection adds graded multi-fault difficulty (\S\ref{sec:cross-app:graded}) precisely to test the system on more production-realistic compound anomalies, but the source-application headline numbers are still single-fault. The cross-application evidence covers one new application, not five, and one new asynchronous-messaging paradigm rather than the full range of paradigms a survey of microservice architectures would enumerate; we mark this as a deliberate scope choice that the cross-application section calls out as a structural-versus-value compatibility result, not as a claim of universal portability.

\subsection{Construct validity}
\label{sec:threats:construct}

The Hit@K and reciprocal-rank metrics implicitly model an engineer who reviews retrieved candidates serially and adopts the first useful one. In practice an experienced engineer may scan all five suggestions in parallel and disambiguate by comparing them; the serial-review assumption is therefore a conservative model of how the cascade's output is consumed. The expected-time-to-diagnose simulation in \S\ref{sec:results:utility} is an explicit back-of-envelope translation under this serial-review assumption and a fixed per-candidate read cost; the simulation should be read as an order-of-magnitude estimate, not as a measurement of engineer productivity. Per-window triage labels are authored by the scenario taxonomy that produced the fault, not by an engineer reviewing the resulting telemetry, so they reflect the \emph{intended} severity rather than an engineer-adjudicated one; the discipline of the bias-avoidance rule and the chronological visibility constraint keep the model from learning labels through telemetry shortcuts, but the labels themselves remain construct-authored. The novelty metric is computed against the cascade's strict gold definition (no past ticket of the same family is visible at the window's timestamp) and treats borderline windows as positives only under the inclusive variant of triage; alternative definitions of \emph{novel} (for instance, a configuration in which any past ticket whose root cause overlaps with the current one would count as a match) would shift the headline by an amount we have not measured.
