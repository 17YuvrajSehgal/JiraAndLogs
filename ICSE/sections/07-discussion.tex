\section{Discussion}
\label{sec:discussion}

\noindent\textbf{Triage-as-classification is the wrong yardstick.} The task most
automated-triage work optimizes---deciding whether a window is ticket-worthy at
all---turns out to be saturated where it is usually measured and barely learnable
where it matters. On injected faults, an active fault is separable from a healthy
baseline almost by construction, so any competent classifier scores nearly
perfectly; on real, engineer-filed tickets, whether an issue is ultimately judged
worth fixing is hardly predictable from its text, and every classifier we tried
sat at chance (\S\ref{sec:results-rq4}). The implication is clarifying: strong
synthetic triage numbers certify the dataset, not the method, and a system tuned
to maximize them optimizes a quantity that does not transfer. The problem worth
solving is the one that does---recognizing \emph{which} past incident a live one
resembles---and it is far from saturated, which is where the gains we report come
from.

\noindent\textbf{Method conclusions flip with scale and realism.} Had we
evaluated retrieval only on the synthetic corpora, as is common, we would have
concluded that fusing retrievers is useless---even mildly harmful---and shipped a
single dense model. On the real corpus, an order of magnitude larger and spread
across two dozen projects, the same fusion is decisively best. The reason is
intuitive in hindsight: in a small, homogeneous memory one retriever dominates and
blending in weaker signals only adds noise, whereas a large, diverse memory
creates exactly the blind spots that complementary retrievers cover. Retrieval
comparisons therefore need to be run at realistic memory scale before their
conclusions can be trusted---one reason we release real-scale data alongside the
synthetic.

\noindent\textbf{Robustness comes from asking ``does this step help \emph{here}?''}
A recurring pattern is that the right action depends on the deployment. The
language-model verifier improves synthetic triage yet is net-negative on real
text and is switched off there; the retrieval-augmented language model never beats
the domain-tuned retriever whose candidates it re-reads. A system that hard-codes
such components pays for them everywhere and is hurt where they misfire. Our
controller instead treats both \emph{what evidence is present} and \emph{whether
an expensive step is actually beneficial} as decisions made per window and per
setting---which is what lets one implementation run from full telemetry down to
bare text, suppress duplicate pages, and shed $40$--$78\%$ of per-window cost with
no loss in accuracy. The practical lesson is that large language models are not a
default win in this domain; the leverage lies in a controller that invokes them
selectively and measures their effect.

\noindent\textbf{The agent's contribution is efficiency and reach, not peak
accuracy.} We are deliberate about not overclaiming. The full agent does not beat
the best single retriever on raw hit rate; it \emph{matches} that quality while
spending a fraction of the budget, and it keeps working when most of the evidence
disappears. In real operations---where instrumentation is uneven, alert volume is
punishing, and compute is metered---peak-quality retrieval when it is warranted, a
lightweight path when it is not, and roughly one alert per incident are precisely
the properties that make a triage assistant deployable. Accuracy that holds only
in the fully instrumented case is of little use for the incident that arrives as a
paragraph of text.

\noindent\textbf{Open problems.} Two gaps frame future work. First, the knowledge
graph supplies correct candidates the other retrievers miss, yet moves top-five
recall little; richer relational extraction or a learned graph matcher, rather
than entity overlap, may convert that latent signal into measured gains. Second,
and more fundamentally, our real-world data is text only while our telemetry-rich
data is synthetic: the open frontier is real telemetry paired with real,
human-written tickets at scale. The datasets, harness, and baselines we release
are meant to lower the barrier to that next step. We discuss the limitations that
temper these conclusions in \S\ref{sec:threats}.
