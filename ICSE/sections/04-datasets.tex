\section{Datasets}
\label{sec:datasets}

Evaluating incident triage faithfully requires data that spans the conditions a
deployed system actually meets---from richly instrumented services, where logs,
metrics, and traces are all available, to bare incident reports with nothing but
text. We assemble three datasets along exactly this spectrum
(Table~\ref{tab:datasets}). Two we \emph{collect and release}: instrumented
microservice systems under controlled fault injection, pairing multi-signal
telemetry with incident tickets and per-window ground-truth labels. The third we
\emph{reuse}: a large corpus of real, engineer-written Apache Jira incidents that
stresses whether methods tuned on synthetic data generalize to the field.

\subsection{Collected Telemetry Datasets}
\label{sec:datasets-synthetic}

\begin{figure}[t]
\centering
\includegraphics[width=\columnwidth]{figures/data-collection.png}
\caption{Data-collection pipeline for the two telemetry datasets. A load
generator exercises an open-source microservice application in Kubernetes while
controlled faults---application-level failures and chaos-engineering network
faults---are injected. The resulting alerts, metrics, logs, cluster events, and
distributed-trace spans are captured per window, and a paired synthetic incident
ticket, rewritten into engineer voice by a language model, is recorded as the
retrieval target. The labelled result is what the triage agent consumes.}
\label{fig:collection}
\end{figure}

Our two datasets are produced by one pipeline (Figure~\ref{fig:collection}): an
open-source microservice application is deployed in a Kubernetes cluster
alongside a full observability stack---Prometheus for metrics, Tempo for
distributed traces, and Loki for logs---and faults are then injected under
controlled, labelled conditions while the platform records everything. The unit of collection is a \emph{scenario}: a
single fault, injected for a fixed duration, with declared ground truth. Each
scenario yields three consecutive observation windows---a fault-free baseline
before injection, the active fault, and a recovery phase after it is
cleared---and every window is labelled ticket-worthy, borderline, or noise. For
each window we archive the raw telemetry it produced: the service's logs, its
CPU, memory, and restart metric series and any firing alerts, the cluster's
events and pod state, and a sample of distributed traces; when a scenario
warrants attention, a synthetic incident ticket is generated and rewritten by a
language model into engineer-voice prose---so that memory tickets read like those
a team would actually file---and stored as the retrieval target. Because injection
is controlled, the resulting window-to-ticket matches are exact ground truth, not
inferred. We deliberately also collect \emph{negatives that
resemble incidents but are not}: near-miss scenarios in which a service is
stressed yet never crosses into failure, and ``orphan'' faults that genuinely
occur but should produce no ticket---both essential for measuring false alarms.

The two applications are chosen to differ in \emph{architecture}, not merely in
scale. \textbf{Online Boutique} is a twelve-service e-commerce demo spanning six
languages; its 27 fault families cover service outages (such as scaling a service
to zero replicas to emulate a hard down), latency and resource-pressure
degradations (including slow memory and connection leaks), pod restarts, network
faults injected with a chaos-engineering tool (partitions, packet loss, DNS
failures), and deployment churn. \textbf{OpenTelemetry Demo} adds architectural
distance: eighteen services across ten languages with a Kafka message bus and
asynchronous downstream consumers. Its 49 scenarios range from single-service
faults---triggered through the application's own built-in failure flags---to
Kafka-specific failures such as broker outages and consumer lag, which have no
Online-Boutique analogue, up to concurrent and cascading multi-fault episodes.
Together the two yield roughly $6{,}700$ and $1{,}600$ labelled windows and tens
of gigabytes of raw telemetry, released as self-contained archives. To our
knowledge no public dataset couples this breadth of operational telemetry with
incident tickets and per-window labels; the release is meant to let others study
triage on realistic, fully observable incidents without rebuilding the collection
infrastructure. Each dataset is partitioned by fault family so that training and
test windows never share a scenario.

\subsection{Real-World Incidents: World of Logs}
\label{sec:datasets-wol}

To test generalization beyond synthetic faults, we reuse the \emph{World of Logs}
collection~\cite{xiao2026worldoflogs}, itself built on a large public archive of
Apache Jira issues~\cite{montgomery2025jira}. This dataset is \emph{not} a
contribution of this paper; we derive an evaluation slice from it. From its
Apache projects we take 24 distributed-systems projects---message brokers,
datastores, stream processors, the Hadoop ecosystem, and others---yielding a
memory corpus of $38{,}642$ resolved bug tickets and $78{,}140$ query incidents.
The queries carry the same three-way labels as our synthetic data, assigned
deterministically from each ticket's type and resolution: a fixed bug is
ticket-worthy, an engaged-but-ambiguous outcome is borderline, and rejected or
non-bug issues are noise.

Two properties make this a demanding test. First, it is \emph{text only}---real
engineer-written descriptions with no telemetry---so it exercises precisely the
degraded, retrieval-driven regime to which our controller must fall back. Second,
we evaluate under a \emph{leave-domain-out} split: two entire project families (a
message broker and a database engine, $13{,}388$ incidents) are held out for
testing, forcing the system to match incidents in projects it never saw during
training. Because real incidents lack a ground-truth ``which past ticket is the
answer'' label, we adopt the collection's inferred matching---a query and a
memory ticket match when they share a project and at least one software
component, with a stricter variant additionally requiring symptom-token
overlap---and report retrieval quality under both relations. As such inferred
labels invite scrutiny, we further validate them with an independent
LLM-as-judge check (\S\ref{sec:results}). Only genuine bug reports carry matches;
borderline and noise queries have none by construction and are excluded from the
retrieval denominators.
